# PRD: 경주시 행정구역 경계 지도 웹앱

## 1. 프로젝트 개요

경상북도 경주시를 대상으로, 동/읍/면 단위 행정구역 경계를 카카오맵 위에 시각화하는 간단한 웹앱을 만든다. 기존에 보유하고 있던 경상북도 전체 행정구역 폴리곤 원본(`gyeongsangbuk-do_adiminisrative_map.json`, EPSG:5186, 이름 정보 없음)은 이미 전처리 스크립트(`scripts/match_gyeongju_regions.py`)를 통해 카카오 로컬 REST API(coord2regioncode)로 행정구역명을 매칭하고, 경주시 폴리곤만 추출하여 WGS84 GeoJSON(`data/gyeongju_dong_boundaries.geojson`)으로 변환해 둔 상태다. 이번 프로젝트의 목적은 이 전처리 결과 데이터를 활용해, 지도 위에 경주시 동/읍/면 경계와 이름을 표시하고 주요 관광/교통 명소 4곳(경주역, 황리단길, 경주월드, 석굴암)에 마커를 표시하는 프론트엔드 웹앱을 신규로 개발하는 것이다. 별도의 서버 로직(DB, API 서버)은 두지 않고, 정적 파일만으로 동작하는 가벼운 웹앱으로 만들어 로컬 실행 및 GitHub Pages 배포까지 지원한다.

## 2. 타깃 사용자 분석

**페르소나 1: 경주 여행을 준비하는 일반 여행객**
- 경주가 처음이라 어느 동네(읍/면/동)에 어떤 명소가 있는지 지리적 감각이 없다.
- "황리단길이 정확히 어느 동에 속하는지", "석굴암이 시내에서 얼마나 먼지" 같은 것을 지도로 한눈에 파악하고 싶어 한다.
- 니즈: 행정구역 경계와 주요 명소 위치를 함께 보면서 이동 동선을 가늠하고 싶음.

**페르소나 2: 지역 행정/관광 데이터를 다루는 실무자·개발자 (본 프로젝트 제작자 자신 포함)**
- 경주시 행정구역 경계 데이터를 시각적으로 검증하고 싶어 함 (전처리한 매칭 결과가 실제 지도 위에서 올바른지 확인).
- 니즈: 폴리곤 매칭 결과를 빠르게 눈으로 검증할 수 있는 간단한 뷰어.

| 행정구역 경계 폴리곤 표시 | `data/gyeongju_dong_boundaries.geojson`을 불러와 각 동/읍/면 경계를 지도 위에 폴리곤(Polygon 오버레이)으로 표시 | P0 (필수) |
| 행정구역 이름 라벨 표시 | 각 폴리곤 영역의 중앙(`center` 좌표, 또는 폴리곤 자체 중심점 계산)에 행정구역 이름을 CustomOverlay로 표시 | P0 (필수) |
| 지도 확대/축소 컨트롤 | 화면 상단에 '+', '-' 버튼과 슬라이더(줌 레벨 조절 UI)를 배치하여 지도 확대/축소 가능 | P0 (필수) |

| 마커 클릭 시 이름 표시 | 마커 클릭(또는 마커 상단 라벨 상시 표시)으로 장소명을 확인할 수 있음 | P1 (있으면 좋음) |
| 반응형(모바일) 레이아웃 대응 | 모바일 브라우저에서도 지도와 컨트롤이 정상적으로 보이도록 레이아웃 대응 | P1 (있으면 좋음) |
| 행정구역 클릭 시 상세 정보 표시 | 폴리곤 클릭 시 해당 동/읍/면의 상세 정보(면적, 인구 등)를 팝업으로 표시 | P2 (향후 확장) |

## 4. 사용자 플로우

1. 사용자가 웹앱 URL(로컬 서버 또는 GitHub Pages 배포 주소)에 접속한다.
2. 페이지 로드 시 카카오맵 JavaScript SDK가 초기화되고, 경주시 전체가 한 화면에 들어오도록 초기 중심좌표/줌 레벨로 지도가 표시된다.
3. 앱은 `data/gyeongju_dong_boundaries.geojson`을 fetch로 비동기 로드한다.
4. 로드된 GeoJSON의 각 feature(동/읍/면)에 대해 카카오맵 Polygon 오버레이를 그리고, `properties.center` 좌표 위치에 CustomOverlay로 행정구역 이름 라벨을 표시한다.
5. 동시에, 사전에 확인해 하드코딩해 둔 4개 명소(경주역, 황리단길, 경주월드, 석굴암) 좌표에 마커를 표시한다.
6. 사용자는 화면 상단의 '+'/'-' 버튼 또는 슬라이더를 조작하여 지도를 확대/축소한다. 확대 시 각 행정구역 경계와 이름, 마커 위치가 지도 축척에 맞춰 자연스럽게 재배치된다.
7. (P1) 사용자가 마커를 클릭하면 해당 장소명이 말풍선(InfoWindow) 형태로 표시된다.
8. (P2, 향후) 사용자가 특정 행정구역 폴리곤 영역을 클릭하면 해당 지역의 상세 정보가 팝업으로 표시된다.

## 5. 기술 스택

- **프론트엔드**: 순수 HTML/CSS/JavaScript (프레임워크, 번들러/빌드 도구 없음)
  - 선택 이유: 사용자가 "간단한 웹앱"을 요청했고, 기능 범위(지도 표시 + 정적 GeoJSON 시각화 + 마커 4개)가 크지 않아 프레임워크 도입에 따르는 복잡도(빌드 설정, 의존성 관리)가 불필요함. GitHub Pages 배포와도 궁합이 좋음(빌드 단계 없이 바로 정적 호스팅 가능).
- **지도 API**: 카카오맵 JavaScript SDK (`kakao_map_JavaScript_APIkey`, `.env`에 기 발급되어 있음)
  - Polygon, CustomOverlay, Marker, InfoWindow 등 카카오맵 JS API 기본 오버레이 기능을 사용.
  - 확대/축소는 카카오맵 기본 컨트롤(`kakao.maps.ZoomControl`)을 사용하거나, 요구사항에 맞춰 커스텀 버튼/슬라이더 UI를 만들고 `map.setLevel()`로 줌 레벨을 제어하는 방식 중 구현 시점에 developer가 UI 요구사항(상단 배치, 슬라이더 형태)에 맞는 방식을 선택.
- **백엔드/서버**: 없음. 정적 파일 서빙 전용 로컬 개발 서버만 사용 (예: `http-server`, `live-server`, Python `http.server` 등 — 구체적 도구는 developer 재량으로 선택하며 프로젝트 코드에 영향 없는 개발 편의 도구로 취급).
  - 선택 이유: 카카오맵 JS SDK는 `file://` 프로토콜에서 정상 동작하지 않을 수 있어 최소한의 정적 서빙이 필요하지만, DB 조회나 별도 API 로직이 필요한 기능이 없으므로 백엔드 애플리케이션 서버는 두지 않음.
- **데이터 저장**: 파일 기반 GeoJSON (`data/gyeongju_dong_boundaries.geojson`), 별도 DB 없음.
- **배포**: GitHub Pages (정적 사이트 호스팅). 배포 작업 자체(git commit/push, GitHub Actions 워크플로 구성 등)는 deploy-manager 서브에이전트가 수행.

**가정 및 확인 필요 사항**:
- 가정: 카카오맵 JS SDK 사용을 위해서는 카카오 디벨로퍼스 콘솔에 서비스 도메인(로컬 개발 시 `localhost`, 배포 후 GitHub Pages 도메인)을 등록해야 하며, 이 도메인 등록 작업은 사용자가 카카오 디벨로퍼스 콘솔에서 직접 수행해야 하는 것으로 가정함 (본 웹앱 코드 범위 밖).
- 가정: 카카오맵 JS 앱 키는 클라이언트 코드에 노출되는 것이 정상적인 사용 방식(도메인 화이트리스트 기반 보안)이므로, `.env` 파일의 값을 그대로 서버에서 숨기지 않고 프론트엔드 JS 코드(또는 별도 config 파일)에 직접 포함시키는 것으로 가정함. `kakao_map_REST_APIkey`는 전처리 스크립트 전용이며 이번 웹앱 런타임에는 사용하지 않음.

## 6. 데이터 구조

### 6.1 행정구역 경계 데이터 (`data/gyeongju_dong_boundaries.geojson`, 기존 파일 재사용)

GeoJSON `FeatureCollection`. 각 `Feature`는 아래 구조를 가짐 (이미 생성 완료, 스키마 변경 없이 그대로 사용):

```json
{
  "type": "Feature",
  "properties": {
    "name": "황남동",
    "sigungu": "경주시",
    "region_code": "4713025000",
    "center": [129.2117, 35.8358]
  },
  "geometry": {
    "type": "Polygon",
    "coordinates": [ ... ]
  }
}
```

- `properties.name`: 동/읍/면 이름 (지도 위 라벨로 표시)
- `properties.sigungu`: 시군구명 (모두 "경주시")
- `properties.region_code`: 행정구역 코드
- `properties.center`: 라벨 표시 기준 좌표, `[경도(lng), 위도(lat)]` 순서 (WGS84)
- `geometry`: WGS84 좌표계로 변환된 폴리곤 경계

### 6.2 주요 명소 마커 데이터 (신규, 웹앱 JS 코드 내 상수로 정의)

```js
const POI_MARKERS = [
  { name: "경주역",   lat: ??, lng: ?? },
  { name: "황리단길", lat: ??, lng: ?? },
  { name: "경주월드", lat: ??, lng: ?? },
  { name: "석굴암",   lat: ??, lng: ?? },
];
```

- **가정**: 위 4개 좌표는 developer 서브에이전트가 구현 시점에 신뢰할 수 있는 출처(카카오맵/네이버맵 등에서 수동 확인)로 정확한 좌표를 확인하여 채워 넣는 것으로 가정함. 특히 **'경주역'은 과거 시내(황남동 인근) 구 역사가 2021년 폐역되고 동해선 개량에 따라 역사가 이전된 이력이 있으므로, 반드시 현재 운영 중인 '경주역'(폐역이 아닌 현 역사)의 좌표인지 명확히 확인 후 하드코딩해야 함**. 이 문서 작성 시점에는 정확한 좌표를 확정하지 않았으며, 임의로 추측한 값을 채우지 않음.
- 런타임에 카카오 장소검색(Keyword) API를 호출하지 않고, 사전에 확인된 고정 좌표를 코드에 직접 포함하는 방식으로 확정함 (사용자 확인 사항).

## 7. 개발 우선순위

**MVP (1차 개발 범위)**
- 카카오맵 JS SDK 연동 및 경주시 중심 초기 지도 표시
- `data/gyeongju_dong_boundaries.geojson` fetch 및 전체 동/읍/면 폴리곤 표시
- 각 폴리곤 중앙에 행정구역 이름 라벨(CustomOverlay) 표시
- 화면 상단 '+'/'-' 버튼 및 슬라이더를 통한 지도 확대/축소 기능
- 경주역, 황리단길, 경주월드, 석굴암 4개 지점 마커 표시 (좌표 사전 확인 후 하드코딩)
- 로컬 정적 서버로 실행 가능한 상태로 개발 완료
- GitHub Pages 배포까지 완료 (deploy-manager 서브에이전트 수행)

**이후 단계 (2차 이후)**
- 마커 클릭 시 InfoWindow로 장소명/설명 표시
- 모바일 반응형 레이아웃 대응
- 행정구역 폴리곤 클릭 시 상세 정보(면적, 인구 등) 팝업 표시
- 마커 대상 명소 추가(사용자 요청 시 확장)
- 행정구역별 색상 구분, 범례 등 시각적 개선
